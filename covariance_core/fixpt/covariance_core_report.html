%PL block v3: STREAMING covariance accumulator - one SCALAR complex
% sample per call, matching real ADC/UART ingest (ingest delivers one
% element at a time) and matching selftest_bram.v's existing native
% output order exactly (no changes needed there).
%
% WHY THE REWRITE (v2 -> v3): v2 took the whole 4x256 frame as one
% function argument. HDL Coder exposed that as a fully parallel port:
% 2048 real values x 16 bit = ~32768 wires, tripping "IO Pin Count 33092
% exceeds threshold 5000" and forcing "4096x faster clock" to squeeze
% 256 MACs into one base-rate tick. Root cause: batch design instead of
% streaming. Fix is MathWorks' documented pattern for exactly this -
% persistent variables as the accumulator (maps to hardware registers),
% called once per sample. Verified bit-identical to the batch X*X'
% computation (max diff 0.0) before shipping.
%
% Protocol: samples arrive in MATLAB X(:) column-major order, i.e. all
% M=4 antenna elements of time-sample 1, then all 4 of time-sample 2,
% etc. (1024 calls with samp_valid=true per 256-sample frame). Buffers
% 4 consecutive scalar arrivals into one antenna-vector, accumulates
% its 4x4 outer product, repeats 256 times, then asserts valid+Rtri.
    persistent R buf pos scount;
Once 
    if isempty(R)
        R = complex(zeros(4,4)); buf = complex(zeros(4,1));
        pos = int32(0); scount = int32(0);
    end

        R(:) = complex(zeros(4,4)); pos(:) = int32(0); scount(:) = int32(0);
    end

25%

            scount(:) = scount + 1;

                    end
                end

            end

        end
    end
